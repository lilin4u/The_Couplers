The Couplers

jenis code smell yang menampilkan adanya pengelompokkan yang ketat (Tight Coupling) antar class/modules atau menampilkan bagaimana bila coupling digantikan dengan delegasi berlebihan (Excessive Delegation).

Tight Coupling menunjukkan bahwa kode dari suatu class atau module memiliki ketergantungan berat terhadap detail internal dari komponen lainnya. Sedangkan Excessive Delegation menunjukkan terdapat banyak lapisan abstraksi atau komponen perantara yang tidak esensial dalam kode.

Walaupun pada umumnya The Couplers tidak akan mempengaruhi jalannya program, namun proses maintenance dan update dapat secara signifikan menjadi lebih sulit untuk dilakukan ke depannya karena adanya dependensi tinggi antar komponen-komponen program sehingga sulit untuk diubah atau dilakukan testing secara terpisah (Test in Isolation).

Terdapat 4 tipe dari The Couplers, yaitu:

• Feature Envy

• Inappropriate Intimacy

• Message Chains

• Middle Man

FEATURE ENVY

Feature Envy adalah code smell dimana suatu method dari suatu class mengakses detail dari object lain secara berlebihan. Dalam kata lain, method tersebut lebih tertarik akan data dan perilaku dari class lain daripada classnya sendiri (cemburu). Hal ini mengindikasikan bahwa method dari class tersebut dapat dipindahkan atau didefinisikan di lokasi yang lebih baik seperti di class yang dicemburui, superclass dari kedua class, atau class lainnya bila perlu.

Contoh :

Dalam case ini, class Student memiliki sebuah method bernama getGPA() yang hanya menggunakan class School untuk melakukan perhitungan. Ini adalah tanda dari Feature Envy karena kelas Student lebih tertarik dengan fitur dari kelas School daripada dengan datanya sendiri.

Refactor :

Sekarang, class Student bertanggung jawab untuk menghitung GPA-nya sendiri, yang merupakan suatu bentuk penerapan yang baik dalam desain berorientasi objek. Aroma Feature Envy telah dihilangkan dengan memindahkan logika perhitungan ke class Student. Hal ini mengurangi tight coupling antara kelas Student dan School.

INAPPROPRIATE INTIMACY

Inappropriate Intimacy adalah code smell dimana dua class memiliki keterkaitan yang erat antara satu sama lain (intimate) sehingga suatu class dapat mengeksploitasi internal field atau method dari class lain. Dalam kata lain, kedua class sangat bergantung pada detail internal satu sama lain, sering kali melalui akses langsung ke data yang bersifat private atau protected.

Contoh :

Pada kasus ini, dapat terlihat adanya keterkaitan yang tidak diperlukan antara class EmployeeEarnings dan TaxCalculator dimana keduanya saling mengakses field internal class (EmployeeEarnings) dan juga mengakses method calculateTax (TaxCalculator) dengan cara yang tidak efektif.

Refactor :

Dengan mengubah TaxCalculator menjadi interface, sekarang class EmployeeEarnings dapat mengkalkulasikan gaji pekerja dipotong pajak tanpa perlu membiarkan class lain mengakses data-data internalnya secara langsung sehingga enkapsulasi pun tetap terjaga dan fleksibilitas cara perhitungan gaji dipotong pajak pun terdukung.

MESSAGE CHAIN

Message Chains adalah code smell dimana ketika ingin mengakses sebuah objek, perlu dilakukan pengaksesan ke objek lainnya melalui pemanggilan dari hasil return method secara terus menerus sehingga membentuk rantai. Dalam hal ini, kita pun menjadi bergantung pada navigasi dari rantai tersebut untuk mengakses suatu objek.

Contoh :

Pada kasus ini, agar dapat mengakses riwayat username dari akun tertentu melalui email memerlukan proses pemanggilan method secara beruntutan. Hal ini membuat penulisan prosesnya menjadi sangat panjang dan sulit dibaca seandainya ada tambahan delegasi.

Refactor :

Sekarang dengan adanya method baru di class UserAccount yang membantu pengambilan riwayat username, proses mengakses riwayat username dari akun tertentu melalui email dapat kita persingkat melalui hide delegation.

MIDDLE MAN

Middle Man adalah code smell dimana suatu class hanya berfungsi sebagai delegator. Dalam kata lain, fungsionalitas dari class tersebut hanyalah mendelegasikan tugas ke class lain sehingga apa kontribusinya terhadap fungsionalitas keseluruhan program?

Contoh :

Pada kasus ini, class Middleman sepenuhnya berfungsi hanya untuk mendelegasikan tugas kepada class Material untuk mendapatkan informasi dan extra dari material. Padahal sebenarnya bisa saja dilakukan langsung oleh class Material tanpa perlunya proses delegasi melalui class Middleman.

Refactor :

Dengan menghapus Middleman kita bisa langsung mengakses info atau extra dari material melalui kedua methodnya sehingga proses akses pun dipersingkat.
